## 2. 架构决策汇总

### 2.1 技术栈选型

| 组件 | 选型 | 版本 | 说明 |
|------|------|------|------|
| **后端框架** | FastAPI | 0.109+ | 异步支持，自动 OpenAPI 文档 |
| **数据库** | PostgreSQL | 16 + pgvector | 业务数据存储 |
| **缓存** | Redis | 7.x | 会话、缓存、限流 |
| **向量检索** | Qdrant | 1.7+ | 四路向量检索 |
| **关键词检索** | Elasticsearch | 8.x | BM25 检索，IK Analyzer 分词 |
| **LLM** | 通义千问 | Qwen-Plus | 阿里云百炼平台 |
| **Rerank 模型** | BGE-Reranker-v2-m3 | 本地部署 | PyTorch + MPS (M1 Mac) |
| **嵌入模型** | BGE-M3 | 本地部署 | 1024 维向量 |

### 2.2 核心架构决策

| 决策点 | 选型 | 备选方案 |
|--------|------|----------|
| **检索引擎** | ES + Qdrant 双引擎 | ES 统一检索 |
| **融合策略** | RRF（k=60） | 加权分数融合 |
| **Rerank 部署** | 本地（PyTorch + MPS） | 云端 API |
| **降级策略** | 三级降级（LLM → Web Search → 规则引擎） | - |
| **Prompt 管理** | Jinja2 模板文件 | 代码内嵌/数据库 |
| **Token 裁剪** | 固定优先级 | 动态裁剪 |
| **会话管理** | Redis（TTL=30 分钟） | 内存会话 |
| **认证方案** | JWT + Redis 黑名单 | Session Cookie |
| **限流策略** | 滑动窗口（Redis ZSet） | 固定窗口/令牌桶 |
| **内容审核** | 先发后审 | 先审后发/机审 + 人审 |
| **成本优化** | 响应缓存（24h TTL） | 小模型过滤 |

### 3.2 RAG 检索流程

```
用户查询："有什么可以用鸡肉和土豆做的简单菜"
              │
              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  Step 1: Query 理解与改写                                                    │
│  - 意图识别：食材搜索                                                       │
│  - 实体抽取：{ingredients: ["鸡肉", "土豆"], difficulty: "简单"}             │
│  - Query 扩展：同义词扩展（鸡肉→鸡/鸡翅/鸡腿）                                 │
│  - 改写后："鸡肉 土豆 简单 快手菜"                                           │
└─────────────────────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  Step 2: 双路召回 (并行执行)                                                 │
│  ┌─────────────────────────┐  ┌─────────────────────────┐                 │
│  │ Elasticsearch (BM25)    │  │   Qdrant (向量检索)      │                 │
│  │ - 关键词匹配            │  │   - name_vec (权重 1.5)  │                 │
│  │ - IK Analyzer 分词      │  │   - desc_vec (权重 1.2)  │                 │
│  │ - TopK=50               │  │   - step_vec (权重 0.8)  │                 │
│  │                         │  │   - tag_vec (权重 1.0)   │                 │
│  │                         │  │   - TopK=50 (每路)       │                 │
│  └───────────┬─────────────┘  └───────────┬─────────────┘                 │
└──────────────┼─────────────────────────────┼───────────────────────────────┘
               │                             │
               └──────────────┬──────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  Step 3: 结果融合 (RRF 倒排融合，k=60)                                       │
│  score = 1 / (k + rank)                                                     │
│  合并两路结果，去重，生成候选集 (约 80-100 条)                                 │
└─────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  Step 4: 重排序 (BGE-Reranker-v2-m3，本地部署)                               │
│  - PyTorch + MPS (M1 Max)                                                   │
│  - 延迟：<100ms/批次                                                        │
│  - 附加权重：用户偏好 x1.2，菜谱热度 x(1+log(fav_count)/100)                │
└─────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  Step 5: 返回 Top10                                                          │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 RAG 生成流程

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   检索结果      │ --> │  Prompt 组装    │ --> │   LLM 调用      │
│   (10 个菜谱)    │     │  (Jinja2 模板)   │     │  (通义千问)     │
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                          │
                                                          ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   用户响应      │ <-- │   后处理        │ <-- │   流式输出      │
│   (推荐结果)    │     │   (格式化)      │     │   (C 端)        │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**三级降级策略：**
```
LLM API 故障 → Web Search → 规则引擎（最终兜底）
```

## 4. 数据模型

### 4.1 核心表（P0）

| 表名 | 说明 | 字段数 |
|------|------|--------|
| `users` | C 端用户 | 10 |
| `recipes` | 菜谱主表 | 16 |
| `recipe_ingredients` | 菜谱 - 食材 | 7 |
| `recipe_steps` | 菜谱 - 步骤 | 6 |

### 4.2 扩展表（P1/P2）

| 表名 | 优先级 | 说明 |
|------|--------|------|
| `favorites` | P1 | 用户收藏 |
| `search_history` | P1 | 搜索历史 |
| `enterprises` | P2 | B 端企业 |
| `standard_recipes` | P2 | B 端标准化配方 |
| `inventory` | P2 | B 端库存 |
| `purchase_orders` | P2 | B 端采购 |

### 4.3 Redis 数据结构

| Key 格式 | 类型 | TTL | 用途 |
|----------|------|-----|------|
| `session:{session_id}` | Hash | 30 分钟 | 会话缓存 |
| `user:profile:{user_id}` | JSON | 24 小时 | 用户画像 |
| `search:cache:{query_hash}` | List | 1 小时 | 搜索缓存 |
| `llm:cache:{query_hash}` | String | 24 小时 | LLM 响应缓存 |
| `ratelimit:sliding:{key}` | ZSet | 动态 | 限流计数 |

### 4.4 Qdrant Collection

| 配置 | 值 |
|------|-----|
| **Collection 名** | `recipes` |
| **向量** | name_vec, desc_vec, step_vec, tag_vec (1024 维，COSINE) |
| **Payload** | cuisine, difficulty, taste, prep_time, cook_time, user_id, enterprise_id, is_public, audit_status |
| **量化** | BinaryQuantization |

## 5. 安全与限流

### 5.1 JWT 配置

| 配置项 | 值 |
|--------|-----|
| **算法** | HS256 |
| **Access Token** | 2 小时 |
| **Refresh Token** | 7 天（Redis 存储） |

### 5.2 限流阈值

| API 类型 | 限流维度 | 阈值 |
|----------|----------|------|
| **搜索 API** | 用户维度 | 30 次/分钟 |
| **菜谱详情** | 用户维度 | 60 次/分钟 |
| **LLM 生成** | 用户维度 | 10 次/分钟 |
| **登录/注册** | IP 维度 | 5 次/分钟 |

### 5.3 错误码分类

| 分类 | 范围 | 说明 |
|------|------|------|
| 客户端错误 | 1001-1999 | 参数错误、认证失败、权限不足等 |
| 服务端错误 | 2001-2999 | 内部错误、数据库错误、缓存错误等 |
| 第三方错误 | 3001-3999 | 限流、LLM 故障、短信服务故障等 |
| 业务错误 | 4001-4999 | 审核未通过、收藏已存在、库存不足等 |

## 6. Harness Agent 协作模式

### 6.1 Sprint 分解

| Sprint | 功能模块 | 验收标准 |
|--------|----------|----------|
| **1** | Elasticsearch 基础搭建 | ES 启动成功，IK 分词测试通过 |
| **2** | Qdrant 向量索引搭建 | 向量检索返回结果，延迟<100ms |
| **3** | 双路召回实现 | 两路召回各自返回 Top50 |
| **4** | RRF 融合实现 | 融合后结果正确，去重完成 |
| **5** | Rerank 服务集成 | Rerank 推理成功，延迟<100ms |
| **6** | 缓存与优化 | 缓存命中率>40%，P99<1s |

### 6.2 数据导入流水线

```
抓取 → 解析 → 清洗 → 校验 → 入库 → 索引
```

**增量导入策略：**
- 批次大小：100 个菜谱/批
- 断点续传：记录已导入文件列表
- 失败重试：单个失败不影响整批

| **搜索 P50 延迟** | < 200ms | 50% 请求的延迟 |
