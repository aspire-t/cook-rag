# CookRAG AI 员工手册

## 6 步标准开发流程

**每次开发必须严格遵守以下流程，缺一不可：**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          CookRAG 标准开发流程                                │
└─────────────────────────────────────────────────────────────────────────────┘

  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ Step 1   │ -> │ Step 2   │ -> │ Step 3   │ -> │ Step 4   │ -> │ Step 5   │ -> │ Step 6   │
  │ 领取任务 │    │ 开发实现 │    │ 测试验证 │    │ 记录日志 │    │ 更新任务 │    │ 提交代码 │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

---

## Step 1: 在 task.json 里领任务

### 1.1 领取任务规则

```bash
# 1. 查看任务列表
cat tasks.json | jq '.[] | select(.status == "pending")'

# 2. 选择任务（优先级：P0 > P1 > P2，Sprint 从小到大）
# 3. 检查依赖是否已完成
# 4. 将任务状态改为 in_progress
```

### 1.2 任务领取检查清单

- [ ] 确认任务优先级（P0 优先）
- [ ] 确认 Sprint 顺序（从小到大）
- [ ] 检查依赖任务是否已完成（`dependencies` 字段）
- [ ] 更新任务状态为 `in_progress`
- [ ] 记录领取时间到 `progress.txt`

## Step 2: 开发实现

### 2.1 代码规范

**Python 代码：**
- 使用类型提示（Type Hints）
- 遵循 PEP 8 风格指南
- 函数不超过 50 行
- 类不超过 200 行

**FastAPI 路由：**
```python
@app.get("/api/v1/{resource}/{id}")
async def get_resource(
    id: UUID,
    current_user: dict = Depends(get_current_user)
) -> APIResponse:
    """
    获取资源详情
    
    Args:
        id: 资源 UUID
        current_user: 当前登录用户
    
    Returns:
        APIResponse: 标准响应格式
    
    Raises:
        ResourceNotFoundError: 资源不存在
    """
    pass
```

**API 响应格式：**
```python
{
    "code": 0,              # 0=成功，非 0=错误码
    "message": "success",   # 人类可读描述
    "data": {...},          # 响应数据
    "meta": {               # 元数据
        "request_id": "uuid",
        "timestamp": 1713513600,
        "latency_ms": 45
    }
}
```

## Step 3: 必须测试和验证

### 3.1 测试类型

**必须执行的测试：**

| 测试类型 | 命令 | 通过标准 |
|----------|------|----------|
| **类型检查** | `mypy app/` | 0 errors |
| **代码风格** | `ruff check app/` | 0 errors |
| **单元测试** | `pytest tests/unit/ -v` | 100% 通过 |
| **集成测试** | `pytest tests/integration/ -v` | 100% 通过 |
| **构建检查** | `python -m py_compile app/main.py` | 无语法错误 |

### 3.2 浏览器实际操作（前端功能）

```bash
# 1. 启动开发服务器
python app/main.py

# 2. 使用 Playwright 自动化测试
# 或使用浏览器手动测试

# 3. 测试检查清单
- [ ] API 返回正确状态码
- [ ] 响应格式符合标准
- [ ] 错误处理正常
- [ ] 认证/授权正常
- [ ] 性能指标达标（P99 < 1s）
```

```

### 浏览器测试
- [ ] 页面加载正常
- [ ] 功能操作正常
- [ ] 错误提示正确

## Step 4: 更新 progress.txt

### 4.1 日志格式

```markdown
## {日期} - Sprint {N} - Task {ID}

**任务**: {任务主题}

**完成内容**:
- 实现了 XXX 功能
- 添加了 XXX 接口
- 修复了 XXX 问题

**测试过程**:
1. 运行类型检查：通过
2. 运行单元测试：10  passed, 0 failed
3. 浏览器实际操作：功能正常

**遇到的问题**:
- 问题 1: {描述}
  解决方案：{方案}

**提交记录**: {git commit hash}
```

```markdown
## 2026-04-21 - Sprint 1 - Task 2

**任务**: 实现数据库 DDL 和数据迁移

**完成内容**:

**遇到的问题**:
- pgvector 扩展需要先 CREATE EXTENSION
  解决方案：在 migration 中先检查扩展是否存在

**提交记录**: abc1234
```

## Step 5: 更新 task.json 标记完成

### 5.1 任务完成检查清单

- [ ] 所有测试通过
- [ ] progress.txt 已更新
- [ ] 代码已格式化（ruff format）
- [ ] 无 TODO/FIXME 遗留

### 5.2 更新任务状态

```bash
# 使用 jq 更新任务状态
jq '(.[] | select(.id == "1") | .status) = "completed"' tasks.json > tasks.json.tmp
mv tasks.json.tmp tasks.json
```

### 5.3 任务结构

```json
{
  "id": "1",
  "subject": "任务主题",
  "description": "任务描述",
  "status": "pending|in_progress|completed|blocked",
  "sprint": "Sprint N",
  "priority": "P0|P1|P2",
  "dependencies": ["1", "2"],
  "completed_at": "2026-04-21",  // 完成时添加
  "commit_hash": "abc1234"       // 完成时添加
}
```

## Step 6: git 提交代码

### 6.1 提交信息格式

```
<type>: <subject>

<body>

<footer>
```

### 6.2 Type 类型

| Type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 实现用户登录` |
| `fix` | Bug 修复 | `fix: 修复 JWT 验证问题` |
| `docs` | 文档更新 | `docs: 更新架构文档` |
| `style` | 代码格式 | `style: 格式化代码` |
| `refactor` | 重构 | `refactor: 优化数据库连接` |
| `test` | 测试 | `test: 添加用户服务测试` |
| `chore` | 构建/工具 | `chore: 更新依赖版本` |

### 6.3 提交信息示例

```bash
feat: 实现数据库 DDL 和数据迁移

- 创建 users 表（用户管理）
- 创建 recipes 表（菜谱主表）
- 创建 recipe_ingredients 表（食材）
- 创建 recipe_steps 表（步骤）
- 添加 GIN 索引和全文搜索
- 配置 Alembic 迁移

测试:
- SQL 语法检查通过
- 数据库迁移测试成功

关联任务：Task #2
```

### 6.4 提交前检查

```bash
# 1. 查看变更
git status
git diff

# 2. 确保没有敏感信息
# - 无 .env 文件
# 无密码/密钥
# 无 credentials.json

# 3. 确保没有大文件
# - 单文件 < 10MB

# 4. 提交
git add <files>
git commit -m "提交信息"
git push origin main
```

## 错误码规范

```python
# 错误码分类
1001-1999: 客户端错误（参数错误、认证失败等）
2001-2999: 服务端错误（内部错误、数据库错误等）
3001-3999: 第三方服务错误（限流、API 故障等）
4001-4999: 业务逻辑错误（审核未通过、收藏已存在等）
```
